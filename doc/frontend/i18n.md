# Frontend Internationalization (i18n)

**Path**: `frontend/src/i18n/`
**Purpose**: Localize the entire app UI. English is the default; Simplified Chinese is selectable from **Settings → Appearance → Language**. Switching language re-renders the whole interface instantly and persists across reloads.

---

## Design Rationale

The app has **no React context providers** — global state lives in the Zustand store and side effects (e.g. theme) are applied in `App.tsx`. Rather than introduce a provider-based library (`react-i18next`), i18n is a **lightweight, zero-dependency layer** that mirrors the existing patterns:

- The selected `language` is a normal **persisted store field**, exactly like `uiTheme`.
- A `useT()` hook subscribes to that field, so any component re-renders the moment the language changes — no provider, no reload.
- The dictionaries are plain typed objects. `en` is the single source of truth, and `zh` is **typed against it** so the compiler enforces key parity.

---

## Architecture

```
Zustand store   →  language: 'en' | 'zh'   (persisted, default 'en', key 'lv-lang')
src/i18n/en.ts  →  nested dictionary — SOURCE OF TRUTH for keys + shape
src/i18n/zh.ts  →  same shape, Simplified Chinese values (typed as Dict)
src/i18n/index.ts → translate / translateN + useT() hook (reads the store)
App.tsx         →  side-effect: document.documentElement.lang = 'zh-CN' | 'en'
SettingsDialog  →  "Language" row in the Appearance (06) section
Components       →  const { t } = useT(); t('namespace.key', vars)
```

| File | Role |

| `frontend/src/i18n/zh.ts` | Simplified Chinese dictionary, typed `Dict` so it must match `en` exactly. |
| `frontend/src/i18n/index.ts` | `translate`, `translateN`, and the `useT()` hook. |

## The Dictionary and `Dict` Type

`en.ts` is a nested, namespaced object declared `as const`. Namespaces mirror component areas: `common`, `settings`, `sidebar`, `chat`, `authLogin`, `deleteAccount`, `plan`, `metrics`, `memory`, `tools`, `skills`, `themeToggle`, `fileViewer`.

```typescript
export const en = {
  sidebar: {
    newChat: 'New chat',
    searchTitle: 'Search (⌘K)',
    noResults: 'No results found',
  },
  metrics: {
    requestsRecorded_one: '{count} request recorded',
    requestsRecorded_other: '{count} requests recorded',
  },
  // …
} as const

// Leaf strings widened to `string`, so translated dictionaries keep the exact
// key shape but supply their own values (not forced to match the en literals).
type Localized<T> = { [K in keyof T]: T[K] extends string ? string : Localized<T[K]> }
export type Dict = Localized<typeof en>
```

`zh.ts` imports `Dict` and annotates the export:

```typescript
import type { Dict } from './en'
export const zh: Dict = { /* same keys, Chinese values */ }
```

**Why this matters:** if a key is added to `en` without a `zh` counterpart (or vice-versa), `tsc` fails. The two dictionaries can never silently drift. Verify with `npx tsc --noEmit`.

## The `useT()` Hook

```typescript
const { t, tn, language } = useT()

t('sidebar.newChat')                          // → "New chat" / "新建对话"
t('chat.chartError', { message: err })        // interpolates {message}
tn('metrics.requestsRecorded', count)         // plural — see below
```

`useT()` reads `language` from the store and memoizes the bound functions. Internally:

- **`translate(lang, key, vars)`** — dot-path lookup (`'a.b.c'`), `{var}` interpolation, with **English fallback**: a missing `zh` value degrades to the English string rather than showing a raw key.
- **`translateN(lang, key, count, vars)`** — selects `${key}_one` vs `${key}_other` and auto-injects `{count}`.

Both are also exported standalone for the rare non-component call site.

### Interpolation

Values use `{name}` placeholders, replaced by the `vars` object:

```typescript
t('tools.removeServerConfirm', { name })   // "Remove MCP server \"{name}\"?"
```

### Pluralization

English plural strings come in `_one` / `_other` pairs; `tn` picks one by `count`. Chinese has no plural inflection, so both keys hold the same value:

```typescript
// en.ts
metrics: { requestsRecorded_one: '{count} request recorded',
           requestsRecorded_other: '{count} requests recorded' }
// zh.ts
metrics: { requestsRecorded_one: '已记录 {count} 个请求',
           requestsRecorded_other: '已记录 {count} 个请求' }
```

## Store, `<html lang>`, and Settings Entry

The `language` field follows the same localStorage-backed pattern as `uiTheme`:

```typescript
// store/index.ts
language: (localStorage.getItem('lv-lang') as 'en' | 'zh') ?? 'en',
setLanguage: (language) => { localStorage.setItem('lv-lang', language); set({ language }) },
// …and `language` is added to the persist `partialize` list.
```

`App.tsx` reflects the choice on the document for accessibility / browser hints:

```typescript
useEffect(() => {
  document.documentElement.lang = language === 'zh' ? 'zh-CN' : 'en'
}, [language])
```

The user-facing control is a **Language** `SettingsRow` in the Appearance section of `SettingsDialog.tsx`, mirroring the Theme radio group (English / 简体中文 → `setLanguage`).

## Conventions

- **Not translated** (left as literals, never routed through `t`): the brand name **`LyndonLLM`**, the mode names **Chat / Cowork / Code**, the typeface name **Inter**, and code-like tokens (`MCP`, `stdio`, `SKILL.md`, URLs).
- **Keyboard glyphs stay literal** inside values: `⌘K`, `↑↓`, `↵`, `esc`, `⌥↵`. Translate the surrounding words only (e.g. `'Search (⌘K)'` → `'搜索 (⌘K)'`).
- **Namespacing**: each component owns a namespace; keys are `camelCase`; nested groups (e.g. `settings.profile.avatar.*`, `authLogin.title.<mode>`) organize larger surfaces.
- **User content wrappers** use a `{value}` placeholder so the interpolated text stays out of the dictionary.

## Adding to the System

**Add a new string:**
1. Add the key under the appropriate namespace in `en.ts`.
2. Add the identical key with a Chinese value in `zh.ts` (omitting it is a compile error).
3. Use it: `const { t } = useT()` → `t('namespace.key')`.

**Add a new language (e.g. `ja`):**
1. Create `frontend/src/i18n/ja.ts` exporting `export const ja: Dict = { … }`.
2. Register it in `index.ts`: extend `Language` to `'en' | 'zh' | 'ja'` and add `ja` to the `DICTS` map.
3. Add the option to the store's `language` union and to the Language `SettingsRow`.

## Verification

```bash
cd frontend
npx tsc --noEmit                                   # enforces en/zh key parity
npx eslint . --ext ts,tsx --max-warnings 0
npx prettier --write "src/**/*.{ts,tsx,css}"
npm run dev                                         # Settings → Appearance → Language
```

Manual checks: default load is English; switching to 简体中文 re-renders sidebar, settings, chat, auth dialogs, and panels instantly; refresh persists the choice (localStorage `lv-lang`) and sets `<html lang="zh-CN">`; brand/mode names and keyboard glyphs remain unchanged; no raw `namespace.key` strings appear (the English fallback covers any gap).
